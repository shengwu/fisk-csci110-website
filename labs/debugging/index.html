Lab 4: Debugging

When your programs run into errors, you'll have to figure out why they broke. This is called debugging.

Anatomy of a stack trace

A stack trace is something that looks like this. It tells you the error that you ran into and where it happened. Then it traces all the way up through the code. For example, if you run into an error calling System.out.println from the main function, it would tell you that the error occurred at System.out.println, which you called from main, etc.


Exception in thread "main" java.lang.NullPointerException
    at com.example.myproject.Book.getTitle(Book.java:16)
    at com.example.myproject.Author.getBookTitles(Author.java:25)
    at com.example.myproject.Bootstrap.main(Bootstrap.java:14)

For more information, see http://stackoverflow.com/questions/3988788/what-is-a-stack-trace-and-how-can-i-use-it-to-debug-my-application-errors

The three types of errors

  1. Compile-time: discovered when compiling your code. Examples: missing brace, trying to add a boolean and a String, using a variable that doesn't exist
  2. Runtime: your code breaks when you try to run it. This usually happens when an Exception is thrown, which usually means that Java ran into an error that it doesn't know how to handle. Examples: trying to divide by zero, using a String that hasn't been initialized (though the compiler will often catch this at compile time)
  3. Logic: your code runs but it's wrong. Example: A program that calculates the area of a triangle returns h * w instead of 0.5 * h * w
